## 1. ゴール（5/7ミーティング決定事項）

求人ボックス・Indeedに掲載中の **人材紹介会社／人材派遣会社** をターゲットに、
パートナーセールスを介さず **直接アプローチ** することで、
リード獲得のキャッシュフロー遅延を解消し、求人ボックス代理店としての売上を最大化する。

| 工程 | 概要 |
|------|------|
| ① 抽出 | 求人ボックス・Indeed・厚労省「人材サービス総合サイト」から会社名／URL／問合せ窓口を取得 |
| ② 絞込 | 製造業＋人材紹介・派遣会社のみフィルタ |
| ③ 生成 | 自社サービスを入れると、業界・企業特性に合わせた文章をAIが自動作成 |
| ④ 送信 | 各社の問い合わせフォームへ自動投稿 |
| ⑤ 集計 | 送信結果・返信率をダッシュボード化 |

## 2. 情報抽出の実現可能性

### 2-1. 求人ボックス（カカクコム社運営）

| 項目 | 状況 |
|------|------|
| 公式API | **非公開**（求人サイト側の取り込み用フィードのみ。閲覧目的のAPIは無し） |
| 取得手段 | (A) HTMLスクレイピング／(B) AIブラウザ自動操作 |
| 取れる情報 | 会社名・所在地・募集職種・自社サイトURL（記載があれば）・求人ボックス内問合せ導線 |
| 難易度 | **★☆☆**（構造はシンプル、Bot対策は軽め） |
| 留意点 | 利用規約上「自動取得」は明示禁止。**自社で得た情報を自社用途で使う**範囲に留める／間隔を空ける／robots.txt遵守。 |

### 2-2. Indeed

| 項目 | 状況 |
|------|------|
| 公式API | 求人を**掲載する側**向け（XML Feed/Employer API）のみ。閲覧側API無し |
| 取得手段 | AIブラウザ自動操作（**スクレイピング単体は厳しい**：CAPTCHA／IPブロック有り） |
| 難易度 | **★★★**（Bot検知が強力。Anthropic Computer Use等で人間挙動を模す必要あり） |
| 留意点 | 法的リスク・速度ともに求人ボックスより高い。**まず求人ボックスで成立させ、Indeedは第2フェーズ推奨**。 |

### 2-3. 厚労省「人材サービス総合サイト」（補助データソース）

| 項目 | 状況 |
|------|------|
| URL | https://jinzai.hellowork.mhlw.go.jp/ |
| 公開性 | **完全公開・公的サイト**（利用規約上の制約が緩い） |
| 取得情報 | 全国の有料職業紹介事業所・労働者派遣事業所の許可番号／会社名／所在地 |
| 難易度 | **★☆☆** |
| 用途 | 求人媒体側の情報と**突合**して「許可番号付きの確実な人材紹介・派遣会社リスト」を構築 |

### 2-4. 推奨技術スタック

| レイヤ | 採用候補 | 理由 |
|--------|----------|------|
| スクレイピング | **Playwright (Node/Python)** | ヘッドレス＋ステルス対応・モダン |
| AIブラウザ操作 | **Anthropic Computer Use (Claude Sonnet 4.6)** | フォーム構造がサイトごとに違う問題に強い |
| 文章生成 | **Claude API (Sonnet 4.6)** + Prompt Caching | 業界別テンプレ＋プロンプトキャッシュでコスト削減 |
| データ管理 | PostgreSQL / Supabase | 送信履歴・重複防止・返信率トラッキング |
| 実行基盤 | クラウドVM（GCP/AWS）+ ジョブキュー | 並列度とIPローテーション制御 |

## 3. 文章自動生成 ＆ フォーム自動送信

### 3-1. 文章生成

- 入力: 自社サービス概要（求人ボックス代理店）／ターゲット情報（会社名・取扱職種・募集規模など）
- 出力: 業界文脈に沿った問合せ文（300〜500字）。**毎回少しずつ表現が変わる**ため、迷惑メール判定を回避しやすい
- コスト: Claude Sonnet 4.6 ＋プロンプトキャッシュで **1社あたり約 0.3〜1.0円**

### 3-2. フォーム自動送信

- 各社のフォームHTMLをAIに読ませ、フィールド対応を自動判定（氏名・会社名・問合せ内容・電話番号 等）
- reCAPTCHA は **2Captcha等の解決サービス**（1件 約0.5円）または手動レビューキューに退避
- 送信成功／失敗ログを保存し、後続のメール送付や手動架電に流用

## 5. 法務・リスクとその対策

| リスク | 内容 | 対策 |
|--------|------|------|
| 利用規約違反 | 各媒体が自動取得を禁止している場合あり | アクセス間隔を空ける／同一IP連投を避ける／結果を社外配布しない |
| 不正アクセス禁止法 | ログイン後画面のスクレイピングはNGの可能性 | **ログイン不要の公開ページのみ**を対象とする |
| 業務妨害／迷惑送信 | 大量フォーム投稿は受領企業のクレーム源 | 1社1回まで、文面を個別最適化、配信停止希望は即除外 |
| 特定電子メール法 | メール送信時はオプトアウト記載必須 | フォーム送信中心とし、メール展開時は法令準拠 |

> **方針**: 公開情報・公的データのみを対象に、節度ある送信頻度で運用すれば実用範囲内で運用可能。

## 9. 実証結果（2026-05-08 追記）

§7 の結論を裏付けるため、**最小規模 PoC を実装・実測**。求人ボックス→AI抽出→厚労省突合まで一気通貫で動作することを確認。

### 9-1. 実装

リポジトリ `98cmd/main`、ブランチ `claude/review-system-requirements-FfPKg`、ディレクトリ `poc/` 配下。

- 求人ボックス検索→詳細ページ取得（Playwright）
- HTML → 構造化データ抽出（Claude Sonnet 4.6 + JSON Schema 出力）
- 人材紹介・派遣判定（AI 推定）
- 厚労省「人材サービス総合サイト」で会社名検索 → 許可番号・住所・電話を補完
- 営業文章生成（Claude Sonnet 4.6 + プロンプトキャッシュ）
- CSV / JSON 出力

### 9-2. 実測（5社サンプル / クエリ「人材紹介 製造」）

| 指標 | 値 | 備考 |
|------|------|------|
| 人材紹介・派遣判定 | **5/5（100%）** | プレックス / タイズ / キチナン / ビータス / タイミー |
| 業種要約品質 | 5/5 高精度 | 各社の事業構成・特化領域を 1〜2 行で正確に把握 |
| 許可番号取得率（求人ボックスのみ） | 1/5（20%） | 求人ボックス詳細ページに表示する企業のみ |
| **許可番号取得率（+ 厚労省補完）** | **4/5（80%）** | 厚労省マスタで唯一マッチしなかったのは「ビータス」（表記揺れ） |
| 住所取得率（+ 厚労省補完） | 4/5（80%） | 同上 |
| 電話番号取得率（+ 厚労省補完） | 4/5（80%） | 同上 |
| 公式サイト URL 取得率 | 1/5（20%） | 求人ボックス詳細にあれば。本格運用は Google 検索補完が必要 |
| 営業文章品質 | 各社個別最適化 | 件名 / 本文構成・CTA「30分情報交換」を控えめに統一・AIっぽさ低 |

### 9-4. 法務面

- 求人ボックスは**ログイン不要の公開ページのみ**を対象。アクセス間隔 5 秒・件数キャップ。
- 厚労省「人材サービス総合サイト」は**公開・公的サイト**。アクセス間隔 3 秒。
- フォーム自動送信は**意図的に未実装**（次フェーズで法務確認後に判断）。

### 9-5. 残課題と次の打ち手

| 課題 | 対策 |
|------|------|
| 公式サイト URL 取得率 20% | 厚労省で許可番号取得 → 会社名で Google CSE 検索 → 公式 URL 補完（追加実装で 80%+ 想定） |
| 厚労省検索の表記揺れ | カタカナ/漢字/英字・全角/半角の各バリエーションでフォールバック検索 |
| 同一会社の別求人重複 | 会社名ベース重複排除を実装済（PoC で検証） |
| LLM URL 幻覚 | extractor 側で punycode ホストへ強制正規化を実装済 |

### 9-6. 結論

第1フェーズ（求人ボックス＋厚労省連携）は実装可能と確認。**コスト・精度ともに § 4 試算の範囲内**で、本開発に進めても予算オーバーリスクは低い。
